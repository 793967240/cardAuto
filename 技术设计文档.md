## 一、引擎选型说明

### 1.1 为什么 Godot 4.x

| 维度 | 评估 |
|------|------|
| 体量适配 | 卡牌 Roguelike 资源量小（图 + 文本 + 数值），Godot 启动快、包体小、与项目体量匹配 |
| 2D 能力 | Godot 2D pipeline 原生强（Control / Node2D / TileMap），水墨 UI 与卷轴时间轴用 Control 即可 |
| 横屏触屏 | Godot Input 系统天然支持触屏，多分辨率适配通过 `stretch_mode = canvas_items` + `aspect = expand` 解决 |
| 数据驱动 | `Resource` + `.tres` 是 Godot 一等公民，卡牌/底座/词条可直接序列化为资源文件，编辑器内可视化编辑 |
| Headless 测试 | `godot --headless` 原生支持，CI 跑战斗模拟器零成本 |
| 开源 / 授权 | MIT，无版税无审核，规避 Unity Runtime Fee 风险 |
| 团队成本 | GDScript 学习曲线短，Python-like，AI 编码友好 |

### 1.2 GDScript vs C# 决策

**主体使用 GDScript**：
- 业务逻辑（UI / 战斗调度 / 数据加载 / 节点交互）全部 GDScript
- 与编辑器集成最深，热重载体验最好
- 移动端打包链路最简单

**保留 C# 升级窗口**：
- 若后期发现战斗模拟器（Headless 跑万局）GDScript 性能不够，**仅把 `BattleSimulator` 模块下沉到 C#**
- 不混用 Node 层；C# 仅作为纯算法库

## 三、核心系统架构

### 3.1 时间轴 Tick 系统（最关键）

#### 数据结构

```gdscript
# src/core/timeline.gd
class_name Timeline extends RefCounted

const TICK_DURATION_SEC := 0.5  # 1 tick = 0.5 秒（来自 GDD §11.1）

var _accumulator: float = 0.0
var _current_tick: int = 0
var _speed_multiplier: float = 1.0  # 1x / 2x / 4x

signal tick_advanced(tick: int)

func update(delta: float) -> void:
    _accumulator += delta * _speed_multiplier
    while _accumulator >= TICK_DURATION_SEC:
        _accumulator -= TICK_DURATION_SEC
        _current_tick += 1
        tick_advanced.emit(_current_tick)
```

#### 链条执行器

```gdscript
# src/core/chain.gd
class_name Chain extends RefCounted

var slots: Array[CardRuntime] = []
var current_index: int = 0
var current_card_progress: int = 0   # 当前牌已经积累的 tick
var recovery_remaining: int = 0      # 修整剩余 tick
var state: ChainState                  # EXECUTING / RECOVERING

func on_tick(ctx: BattleContext) -> void:
    if state == ChainState.RECOVERING:
        recovery_remaining -= 1
        if recovery_remaining <= 0:
            _restart_chain(ctx)
        return

    var card := slots[current_index]
    current_card_progress += 1
    if current_card_progress >= card.effective_cost(ctx):
        card.fire(ctx)               # 卡牌效果在此触发
        current_card_progress = 0
        current_index += 1
        if current_index >= slots.size():
            _enter_recovery(ctx)
```

#### 设计要点

| 要点 | 实现策略 |
|------|---------|
| 1 tick = 0.5 秒 | 常量定义在 `Tuning` 资源中，便于统一调整 |
| 战斗加速 1x/2x/4x | `_speed_multiplier` 改值即可，不影响逻辑 tick 数 |
| 玩家不可被打断（GDD §3.4） | 玩家 `Chain.on_tick` 永不被外部取消，敌人能力只能改 `effective_cost` 或施加未来 debuff |
| 打断效果（GDD §3.5） | `card.fire()` 中调用 `enemy_chain.reset_current_card_progress()`，并施加 `打断免疫` 状态 |
| 修整时长可被改 | `_enter_recovery` 时调用 `ctx.compute_recovery_duration()`，硬下限 1 tick |

### 3.2 战斗调度器

```
BattleScene (UI)
    ↓ 持有
BattleController (协调器)
    ↓ 持有
BattleContext
    ├── Timeline           （tick 心跳）
    ├── PlayerCombatant    （玩家战斗者）
    │     └── Chain
    └── Array[EnemyCombatant]
          └── Chain (each)

每 frame:
  BattleController._process(delta) → Timeline.update(delta)
  Timeline.tick_advanced → BattleController._on_tick()
    → player.chain.on_tick(ctx)
    → for enemy in enemies: enemy.chain.on_tick(ctx)
    → 检查胜负
    → emit signal 给 UI 更新
```

**关键原则**：
- `BattleController` 在 `core/` 中，**不引用任何 UI 节点**
- UI 通过监听 `event_bus` 信号被动更新
- 这使得 `BattleSimulator`（Headless）可直接驱动 `BattleController` 跑战斗，无需场景树

### 3.3 数据驱动方案

#### CardData 资源

```gdscript
# src/data_models/card_data.gd
class_name CardData extends Resource

@export var id: StringName
@export var display_name: String
@export var cost: int                          # 基础 cost（tick）
@export var card_type: CardType                # ATTACK / DEFENSE / BUFF / CONTROL / SUMMON / SPECIAL
@export var tags: Array[StringName]            # [&"fire", &"sword", &"charge"]
@export var rarity: Rarity
@export_multiline var description_template: String
@export var effect_script: GDScript            # 指向具体效果脚本
@export var icon: Texture2D
@export var consumable: bool = false
@export var upgrade: CardData = null           # +版本（GDD §9）
```

每张卡牌一个 `.tres` 文件，例：

```
data/cards/sword/zhan.tres                # 斩
data/cards/sword/zhan_plus.tres           # 斩+
data/cards/sword/jian_jue.tres            # 剑诀
```

#### 卡牌效果脚本（策略模式）

```gdscript
# src/core/effects/effect_attack.gd
class_name EffectAttack extends CardEffect

@export var damage: int
@export var hits: int = 1

func fire(ctx: BattleContext, source: Combatant) -> void:
    for i in hits:
        var target := ctx.choose_target(source)
        target.take_damage(damage, source.tags)
```

**优势**：
- 美术 / 策划可在 Godot 编辑器中直接配卡，不需碰代码
- `effect_script` 是 `GDScript` 引用，新增效果只需写一个新脚本
- 测试时可 mock effect 验证调用

### 3.4 状态/标签系统

```gdscript
# src/core/status.gd
class_name StatusInstance extends RefCounted

var status_id: StringName       # &"weakness" / &"vulnerable" / &"burn" / &"frozen"
var stacks: int
var remaining_ticks: int        # -1 表示永久
var on_apply: Callable
var on_tick: Callable
var on_expire: Callable
```

`Combatant.statuses: Array[StatusInstance]`，每 tick 遍历调用 `on_tick`。

## 四、UI 与输入

> EA 走 Steam，PC 键鼠为首发主交互。本章重写以 PC 为基准。

### 4.1 窗口与显示模式

`project.godot`：

```ini
[display]

window/handheld/orientation="landscape"     ; 安卓自测包用
window/stretch/mode="canvas_items"
window/stretch/aspect="expand"               ; 21:9 / 16:10 等扩展显示
window/stretch/scale_mode="fractional"      ; 平滑缩放
```

**窗口模式（玩家可切换，存配置）**：
- 窗口模式（默认，1280×720 起步）
- 全屏窗口 / 独占全屏
- 边框窗口（多显示器主播首选）

**横屏布局保持**，但 PC 下不强制锁定纵横比，UI 锚点扩展时优先扩到两侧空白区，**不拉伸卡牌本体**。

### 4.2 多分辨率适配策略

| 分辨率 | 优先级 | 策略 |
|--------|------|------|
| 1920×1080 (16:9) | 必须 | 设计基准，最优显示 |
| 2560×1440 (16:9) | 必须 | 整数倍 / 高质纹理 / 矢量 UI |
| 3840×2160 (16:9) | 必须 | 矢量 UI 缩放，卡牌图必须 ≥1024 长边 |
| 1280×800 (16:10) | 必须 | Steam Deck，UI 元素安全区收缩 |
| 1920×1200 (16:10) | 推荐 | 同 16:9 处理上下扩展 |
| 2560×1080 (21:9) | 推荐 | UI 锚定不拉伸卡牌；扩展为更长卷轴 |
| 3440×1440 (21:9) | 推荐 | 同上 |
| Android 720×1280 / 1080×1920 | 兼容 | 仅自测包；启用 SafeArea + 强制横屏 |

**关键策略**：
- UI 元素优先用矢量（`Theme` + 9-patch），高 DPI 不糊
- 卡牌图最长边 ≥1024 像素（4K 下不糊）
- `canvas_items + expand` 让 21:9 用户获得更长的时间轴卷轴（额外卡槽不会扩，但可视空间更舒展）
- 字体使用矢量字体 (.ttf)，按显示分辨率动态计算字号

### 4.3 输入抽象层（Input Router）

PC 与触屏使用统一的高层语义事件，由 Router 翻译：

```gdscript
# src/input/input_router.gd
extends Node

# 高层语义事件（订阅这些，不直接订阅原始 Input）
signal pointer_moved(world_pos: Vector2)
signal pointer_pressed(world_pos: Vector2, button: int)   # button: 0=主, 1=次
signal pointer_released(world_pos: Vector2, button: int)
signal pointer_dragged(from: Vector2, to: Vector2, button: int)
signal hover_entered(node: Node)                          # PC only, 触屏不发
signal hover_exited(node: Node)
signal action_triggered(action: StringName)              # 来自快捷键的语义动作

# 输入设备识别（用于 UI 提示切换图标：鼠标 / 手柄 / 触屏）
var current_device: Device = Device.MOUSE_KEYBOARD
signal device_changed(new_device: Device)

func _input(event: InputEvent) -> void:
    if event is InputEventMouseMotion:
        _switch_device(Device.MOUSE_KEYBOARD)
        pointer_moved.emit(event.position)
    elif event is InputEventMouseButton:
        _emit_pointer_event(event)
    elif event is InputEventScreenTouch or event is InputEventScreenDrag:
        _switch_device(Device.TOUCH)
        _emit_pointer_event_from_touch(event)
    elif event is InputEventJoypadButton or event is InputEventJoypadMotion:
        _switch_device(Device.GAMEPAD)
        _emit_gamepad_event(event)
    elif event is InputEventKey and event.pressed:
        _emit_action_from_key(event)
```

**设计要点**：
- UI 节点只监听语义事件，对设备类型无感
- `current_device` 切换时 UI 自动切换提示图标（"按 A 确认" vs "点击确认" vs "按 Enter 确认"）
- 鼠标悬停事件**仅在键鼠模式下**emit，触屏时退化为长按

### 4.4 拖拽控制器（统一处理键鼠与触屏）

```gdscript
# src/input/drag_controller.gd
extends Node

signal card_picked_up(card: CardView)
signal card_dropped(card: CardView, target: SlotView)

const LONG_PRESS_THRESHOLD := 0.4   # 触屏长按阈值
var _picked_card: CardView
var _press_start_time: float

func _ready() -> void:
    InputRouter.pointer_pressed.connect(_on_press)
    InputRouter.pointer_dragged.connect(_on_drag)
    InputRouter.pointer_released.connect(_on_release)
    InputRouter.hover_entered.connect(_on_hover)   # 仅 PC

func _on_hover(node: Node) -> void:
    # PC 鼠标悬停 → 立即弹详情
    if node is CardView:
        _show_detail_popup(node)

func _on_press(world_pos: Vector2, button: int) -> void:
    var card := _hit_test(world_pos)
    if not card: return
    _picked_card = card
    _press_start_time = Time.get_ticks_msec() / 1000.0
    if InputRouter.current_device == Device.TOUCH:
        # 触屏：启动长按定时器作为详情 fallback
        get_tree().create_timer(LONG_PRESS_THRESHOLD).timeout.connect(
            func(): if _picked_card == card: _show_detail_popup(card)
        )
```

## 五、存档系统

> 纯单机买断，无账号系统。本地存档为主，**Steam Cloud 同步**作为跨设备保障。

### 5.1 Run 内存档（每个节点结束自动存）

```gdscript
# src/meta/save_system.gd
const RUN_SAVE_PATH := "user://run.save"

func save_run(state: RunState) -> void:
    var data := {
        "version": 2,
        "character": state.character_id,
        "act": state.act,
        "node_index": state.node_index,
        "hp": state.hp,
        "gold": state.gold,
        "deck": state.deck.map(func(c): return c.id),
        "slots": state.slots.map(func(s): return s.serialize()),
        "relics": state.relics.map(func(r): return r.id),
        "checksum": _compute_checksum(state),     # 防存档损坏
    }
    var file := FileAccess.open_compressed(RUN_SAVE_PATH, FileAccess.WRITE,
                                            FileAccess.COMPRESSION_GZIP)
    file.store_var(data)
```

### 5.3 Steam Cloud 同步

**为什么纯单机也建议开**：
- 玩家换机器 / 重装 / Steam Deck 与 PC 互切，存档跟着走
- 防止本地存档损坏的兜底
- Steam 玩家**默认期望**有云存档（不开会被差评）

**配置**（在 Steamworks 后台 + 项目内）：

```
SteamCloud auto-sync paths:
  - {user_dir}/run.save           ; 当前 Run（最高优先级，覆盖最近）
  - {user_dir}/meta.save          ; 元进度
  - {user_dir}/settings.cfg       ; 玩家设置

冲突策略: keep newer (按本地修改时间)
配额: 100 MB / 100 files（远超需求）
```

**Godot 集成**：使用 GodotSteam 插件（社区活跃，支持 Godot 4.x）。

### 5.4 存档损坏恢复

```gdscript
func load_run() -> RunState:
    if not FileAccess.file_exists(RUN_SAVE_PATH):
        return null
    var data = _load_compressed(RUN_SAVE_PATH)
    if not _verify_checksum(data):
        push_warning("Save corrupted, attempting backup recovery")
        return _try_load_backup()
    return _migrate_if_needed(data)
```

每次存档时**先写到 `run.save.tmp`，校验通过后再原子 rename**，避免半写损坏。
保留最近 3 个备份：`run.save.bak1` / `bak2` / `bak3`。

### 5.5 版本迁移

每次存档 schema 变更 `version` +1，加载时按版本号跑 `_migrate_v1_to_v2()`。
**EA 阶段必须**：每个版本上线前测试旧存档加载（保留前一版本的存档样本作为测试 fixture）。

## 六、性能预算（PC 主，安卓自测附带）

### 6.1 PC 性能目标（首发标杆）

| 指标 | 推荐配置目标 | 最低配置目标 |
|------|------------|-----------|
| 帧率 | 1080p / 144 FPS | 1080p / 60 FPS |
| 战斗场景 Draw Call | < 100 | < 150 |
| 内存占用 | < 1 GB | < 1.5 GB |
| 显存占用 | < 500 MB | < 800 MB |
| 启动时间 | < 3 秒至主菜单 | < 6 秒 |
| 战斗加载 | < 1 秒 | < 2 秒 |
| 4x 加速时帧率 | 不掉帧 | ≥ 30 FPS |

### 6.2 推荐 / 最低配置（Steam 商店页填这个）

| 项 | 推荐配置 | 最低配置 |
|------|---------|---------|
| OS | Windows 10/11 64bit | Windows 10 64bit |
| CPU | Intel i5-8400 / AMD Ryzen 5 1600 | Intel i3-6100 / AMD FX-6300 |
| 内存 | 8 GB | 4 GB |
| GPU | GTX 1050 / Radeon RX 560 / 集显 Iris Xe | GTX 750 / 核显 HD 4000 |
| 存储 | 2 GB | 2 GB |
| 网络 | 不需要 | 不需要 |

### 6.3 Steam Deck 目标

- Verified 认证标 - 必达
- 1280×800 / 60 FPS 稳定
- 全手柄操作完整可用
- 文字大小在 800p 屏幕上可读
- 启动后无需任何键鼠交互即可开玩

### 6.4 安卓自测包目标

低优先级，能跑即可：
- 中端机型（骁龙 7 系列） / 60 FPS
- 内存 < 600 MB
- 主要用于触屏交互和移动端体验快速验证

### 6.5 关键优化点

- 卡牌图集合并（每个角色一张 atlas，4096×4096）
- 字体子集化（中文常用 3500 字 + 卡牌专用术语，约 4500 字）
- 矢量 UI 优先 / 9-patch / Theme，避免大量 raster 图
- `Tween` 复用，禁止每 tick `new()`
- 战斗 `_process` 内零分配（`Object.new()` 移到初始化）
- 动画用 `AnimationPlayer` 不用脚本逐帧改属性
- 渲染后端：**Forward+** (PC 标配) → 性能不足时回退 **Mobile** 后端

## 九、关键技术风险与对策

| 风险 | 严重度 | 对策 |
|------|------|------|
| GodotSteam 与 Godot 版本不兼容 | 高 | 锁定 Godot 4.6 + GodotSteam 对应版本；升级前充分测试 |
